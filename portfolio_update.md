# Portfolio Update — Week 12

**For:** Hiring Manager reviewing the Week 10/11 work
**Context:** Week 12 was a structured peer-review and pedagogy sprint on top of the Tenacious Conversion Engine (B2B sales AI agent) and tenacious-bench (benchmark and SimPO adapter) shipped in Weeks 10 and 11.

## What Was There Before Week 12

The Week 10/11 portfolio contained a working sales agent pipeline, a benchmark system with 200 SimPO preference pairs, an ablation study claiming SimPO is Pareto-optimal at 82% capacity_honesty for 73% lower cost than the constrained-prompt baseline, and an LLM-judge evaluation system (Automaton Auditor) that scores repositories on a 5-point scale.

The system worked. The claims were directionally correct. But several of them could not be defended mechanically when interrogated under pressure.

---

## Five Grounding Commits

**Commit 1 — Architecture description corrected (README.md, method.md)**

The pipeline was documented as an agent using tools. After tracing the actual API calls, this was a category error: `generation/service.py` sends no `tools` parameter, and the model is never given tool schemas in the main pipeline. The README now uses the three-pattern framework (model-as-planner, orchestrator-as-planner/model-as-transform, model-as-reviewer) and describes the model's actual roles accurately: a content rewriter and a governance gate. The proactive tool-use instruction in `method.md` is now scoped to the tau2-bench evaluation context where it applies, and noted as irrelevant to the production pipeline where it does not.

*What this shows:* ability to trace a system at the API level and correct documentation that overstates the model's role — a common failure in shipped LLM systems.

**Commit 2 — Intent classification gap documented (orchestration/handoff.py, method.md)**

`route_inbound_message()` routes entirely by token matching. An opt-out message containing none of the six hardcoded token lists routes to the scheduling branch and books a discovery call. The commit adds a `classify_reply_intent` design spec to `method.md` describing the function-calling tool that would read `message.body` before routing runs, including the tool schema and how the returned label gates the pipeline.

*What this shows:* ability to identify where a production pipeline makes decisions without model input that should involve model input, and to specify the fix at the API level.

**Commit 3 — 82% capacity_honesty claim scoped (methodology.md)**

The Pareto-optimality claim now includes a scope statement: 82% reflects in-distribution performance on template-matched eval data generated by the same pipeline as the training set. The claim is not retracted — it is bounded. A paraphrase robustness eval is documented as planned for v2: same 82 unique input contexts rewritten so escalation and hard-commitment are expressed with different vocabulary. The Pareto-optimality argument is stated as holding within the template distribution, not as a general deployment claim.

*What this shows:* ability to distinguish between a result that is wrong and a result that is correctly bounded — and to add precision to a claim without walking it back.

**Commit 4 — Audit score reporting updated (README.md, audit output)**

The Automaton Auditor now reports `mean ± 95% CI` across multiple runs instead of a single point estimate. The README documents that a single-run score carries approximately ±0.5 uncertainty at 95% confidence given the observed SD of 0.611, and that 9 runs is the minimum for a stable mean. The 15-run dataset already collected is sufficient; the fix was aggregation, not more data.

*What this shows:* ability to recognize when a reporting format implies precision the underlying system does not have, and to fix it with the correct statistical framing.

**Commit 5 — Paired bootstrap choice defended (methodology_rationale.md)**

`run_ablations.py` had implemented paired bootstrap correctly but `methodology_rationale.md` did not explain why pairing was chosen or what it would mean for the 3pp Delta A target if it had not been used. The commit adds a one-paragraph explanation: at task-level correlation of 0.97 and n=59 tasks, the unpaired CI includes zero (target undetectable) and the paired CI excludes zero (target detectable). The variance reduction formula `Var(paired) = Var(A) + Var(B) - 2·Cov(A,B)` is stated and the covariance term is connected to the task-level difficulty structure of the benchmark.

*What this shows:* ability to defend a statistical method choice from first principles, not just cite that it was used.

## Summary

The Week 10/11 portfolio went through five targeted edits. None of them changed what was built. All of them changed whether the claims attached to what was built can be defended in a code review, a research conversation, or a production post-mortem. The system is the same. The account of the system is now accurate.
